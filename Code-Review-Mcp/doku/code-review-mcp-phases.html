code-review-mcp v7 — Zeitliche Einordnung

Wann kann was eingesetzt werden?

Nicht jedes Tool passt in jede Phase. Diese Übersicht zeigt welche Analysen vor, während und nach der Implementierung sinnvoll sind — und warum.

Nach Implementierung

Nach Fertigstellung

Nach Tests, Build und Merge. suggest_boyscout_actions aggregiert Compiler-Gate + Top-5-Findings pro geänderter Datei in einem Call. detect_untested_public_api läuft rein statisch ohne Test-Run; nur die Coverage-Tools (analyze_coverage, analyze_test_health) brauchen einen vorherigen Test-Run.

① Planungsphase
index_project + find_in_index

Vollständige Karte des Projekts: alle Klassen, Services, Interfaces, Routen, Dependency-Graph.

Warum in der Planung?

Bevor du einen neuen Service schreibst, weißt du: Gibt es schon etwas Ähnliches? Welche Interfaces muss er implementieren? Wer wird ihn nutzen? Welche Layer-Regeln gelten?

analyze_type_graph

Vollständiger Typ-Graph: Vererbung, Zyklen, Layer-Violations, meistverwendete Typen.

Warum in der Planung?

Zeigt wo der neue Code andocken muss und welche Zyklen er nicht erzeugen darf. Verhindert Architekturbrüche bevor sie entstehen.

suggest_class_splits

LCOM + Method-Cluster → zeigt welche bestehenden Klassen zu groß sind und wie sie geschnitten werden sollten.

Warum in der Planung?

Wenn du eine bestehende Klasse erweitern willst: Zuerst prüfen ob sie überhaupt noch erweiterbar ist oder ob ein Split vor dem Feature sinnvoller wäre.

analyze_refactoring_safety

Wie viele Dateien nutzen eine Methode? Template-Bindings? Interface-Kontrakt? Risiko-Bewertung.

Warum in der Planung?

Bevor du eine bestehende API änderst: Prüfe wie viele Stellen brechen würden. Lieber jetzt wissen als beim Merge.

analyze_maintainability_index

MI-Score + LCOM für bestehende Klassen die du anfassen wirst.

Warum in der Planung?

MI F + LCOM hoch = Klasse ist schwer zu erweitern. Das beeinflusst die Schätzung und ob ein Refactoring-Ticket davor nötig ist.

analyze_dataflow

Wie fließen Daten zwischen bestehenden Klassen? Nullable-Grenzen, unawaited Tasks.

Warum in der Planung?

Zeigt wo der neue Code in bestehende Datenflüsse eingreift und welche Null-Checks er mitbringen muss.

② Implementierungsphase
review_code / review_file

LLM-Review auf frisch geschriebenem Code — SOLID, Security, Performance, Best Practices.

Warum während der Impl.?

Sofortiges Feedback auf den Code den du gerade schreibst. Wie ein Senior-Reviewer der direkt neben dir sitzt.

💡 Auch: review_code für einzelne Methoden
review_git_diff

Review aller geänderten Dateien seit letztem Commit — staged oder unstaged.

Warum während der Impl.?

Pre-Commit-Hook oder manuell vor dem Push. Fängt Probleme bevor sie im PR landen.

analyze_nullability

Null-Safety in frisch geschriebenem Code: !, .Result, FirstOrDefault ohne Check, subscribe ohne error.

Warum während der Impl.?

Null-Bugs sind die häufigsten Runtime-Crashs. Besser jetzt finden als im Produktionsbetrieb.

analyze_complexity

Zyklomatische Komplexität während du eine Methode schreibst.

Warum während der Impl.?

CC > 10 während der Implementierung ist ein Signal zum Aufteilen — nicht erst beim Review. Frühzeitig extrahieren ist einfacher.

generate_auto_fixes

Konkrete Before/After-Fixes: @Input→input(), *ngIf→@if, .Result→await, CancellationToken.

Warum während der Impl.?

Direkt anwendbar während du schreibst. Neue Patterns lernen durch konkrete Beispiele im eigenen Code.

analyze_control_flow

Unreachable Code, fehlende Return-Pfade, always-true Conditions, Infinite-Loop-Risiken.

Warum während der Impl.?

Logikfehler im Kontrollfluss sind am einfachsten zu fixen während die Methode noch frisch im Kopf ist.

analyze_dead_code

Private Methoden/Felder die nie genutzt werden, unused Imports.

Warum während der Impl.?

Beim Refactoring entstehen oft Reste. Lieber sofort entfernen als technische Schulden anhäufen.

analyze_ast_only

Schnelle Strukturmetadaten ohne LLM-Kosten — Klassen, Methoden, Deps, SOLID-Flags.

Warum während der Impl.?

In CI/CD-Pipelines: schnelles Structural-Gating ohne API-Latenz. Blockiert den Build bei kritischen Verletzungen.

③ Nach der Implementierung
analyze_coverage

Parst lcov.info / cobertura.xml — Line/Branch/Function-Coverage, Grade A–F, uncovered Methoden.

Warum erst danach?

Braucht einen abgeschlossenen Test-Run mit Coverage-Flag. Erst nach "ng test --code-coverage" oder "dotnet test --collect" aufrufbar.

⚠️ Benötigt: Test-Run mit Coverage
detect_untested_public_api

Listet public-Symbole (Methoden, Properties, Get/Set-Accessoren) ohne erkennbare Test-Referenz — Heuristik, kein Test-Run. reason: no_test_file / no_reference_found.

Warum erst danach?

Erster Check nach der Implementierung: zeigt neu hinzugekommene API ohne Test, bevor mit analyze_test_quality die Güte der vorhandenen Tests bewertet wird. Statischer Proxy, ersetzt keinen echten Coverage-Report.

analyze_test_quality

Statische Test-Analyse: no-assertions, tautology, mock-heavy, happy-path-only, fehlende Test-Dateien.

Warum erst danach?

Tests werden meist am Ende geschrieben. Erst wenn Tests existieren macht die Qualitätsanalyse Sinn. Kann aber auch laufend genutzt werden.

analyze_test_health

Dashboard: Coverage + Test-Qualität kombiniert in einem Bericht.

Warum erst danach?

Das vollständige Bild — was ist abgedeckt, was ist gut getestet, was fehlt — ergibt sich erst nach der vollständigen Implementierung.

⚠️ Benötigt: Test-Run mit Coverage
analyze_duplicates

Strukturelle Duplikate via AST-Hash — gleiche Logik in verschiedenen Klassen.

Warum erst danach?

Duplikate entstehen oft durch Copy-Paste während der Implementierung. Nach dem Feature-Freeze sieht man das vollständige Bild.

review_with_index

Review mit vollem Projekt-Kontext — LLM versteht Dependency-Graph bevor es den Code liest.

Warum erst danach?

Nach der Implementierung ist der Index aktuell und enthält den neuen Code. Das gibt dem LLM das präziseste Gesamtbild für den abschliessenden Review.

review_files_batch

Alle geänderten Dateien des Features auf einmal reviewen.

Warum erst danach?

Sprint-End oder vor dem Merge: vollständiger Review aller Files des Features als letztes Qualitätsgate.

analyze_advanced_all

Alle 7 Advanced-Analysen in einem Aufruf: Komplexität + Dead Code + Nullable + Duplikate + Refactoring + AutoFixes + Dataflow.

Warum erst danach?

Der vollständige Gesundheitsbericht für das gesamte Projekt. Ideal nach einem Sprint oder Release als Qualitätsbaseline.

💡
Die wichtigste Erkenntnis
25 von 27 Tools brauchen keinen Build und keinen Test-Run — sie arbeiten rein auf dem Quellcode (AST). Nur die beiden Coverage-Tools benötigen einen vorherigen Test-Run. Das bedeutet: der grösste Teil der Qualitätssicherung kann ohne CI/CD-Pipeline, ohne Build-Umgebung und ohne Wartezeit direkt im Editor oder Pre-Commit stattfinden.
